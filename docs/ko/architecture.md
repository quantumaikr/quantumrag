# 아키텍처

> QuantumRAG가 문서를 처리하고 질문에 답변하는 방식

QuantumRAG는 **Index-Heavy, Query-Light** 철학을 따릅니다. 비용이 큰 연산은 문서 인제스트 시점에 한 번 수행하고, 쿼리 시점에는 빠르고 정확한 검색과 답변 생성을 수행합니다.

---

## 전체 구조

```
                          ┌──────────────────────────────────────┐
                          │            QuantumRAG Engine          │
                          │           (단일 진입점)               │
                          └─────────┬────────────────┬───────────┘
                                    │                │
                     ┌──────────────▼──┐      ┌──────▼──────────────┐
                     │ 인제스트 파이프라인│      │  쿼리 파이프라인     │
                     │  (Index-Heavy)   │      │   (Query-Light)     │
                     └──────────────┬──┘      └──────┬──────────────┘
                                    │                │
              ┌─────────────────────▼────────────────▼─────────────────┐
              │                    스토리지 레이어                       │
              │   SQLite (문서)    LanceDB (벡터)    Tantivy (BM25)     │
              └────────────────────────────────────────────────────────┘
```

## 인제스트 파이프라인

인제스트 파이프라인은 원시 문서를 검색에 최적화된 다층 인덱스로 변환합니다.

### 단계별 흐름

```
문서 (PDF, DOCX, HWP, XLSX, ...)
  │
  ▼
1. 파싱 ───────────── 형식별 파서 (9가지 형식)
  │
  ▼
2. 청킹 ───────────── 전략 자동 선택 (structural → semantic → fixed)
  │
  ▼
3. 프로파일링 ──────── 문서 시그널 추출 (도메인, 구조, 언어)
  │
  ▼
4. 프리앰블 ───────── 청크별 LLM 생성 컨텍스트 접두사
  │
  ▼
5. 팩트 추출 ──────── 규칙 기반 도메인 팩트 추출 (6개 도메인)
  │
  ▼
6. 품질 검증 ──────── 파싱 품질 확인 및 필터링
  │
  ▼
7. 4-Level Indexing
  │  ├─ Level 1: Multi-Resolution 요약 (문서 → 섹션 → 청크)
  │  ├─ Level 2: Structured Fact 인덱스 (엔티티, 속성, 관계)
  │  ├─ Level 3: Derived Index 보강 (동의어, 계층 용어 → BM25)
  │  └─ Level 4: Entity-Centric 역인덱스 (entity → chunk_ids)
  │
  ▼
8. Triple Index 빌드
  │  ├─ Original Embedding (text-embedding-3-small)
  │  ├─ HyPE Embedding (가상 질문 → 임베딩)
  │  └─ Contextual BM25 (Kiwi 형태소 분석 토큰)
  │
  ▼
9. Chunk Constellation Graph (형제, 부모, 엔티티 엣지)
```

### 파싱

형식별 파서 지원:

| 형식 | 확장자 | 파서 |
|------|--------|------|
| 텍스트 | `.txt`, `.text`, `.log` | PlainTextParser |
| Markdown | `.md`, `.markdown` | MarkdownParser (YAML 프론트매터 추출) |
| HTML | `.html`, `.htm` | HTMLParser (태그 제거, 정규화) |
| CSV/TSV | `.csv`, `.tsv` | CSVParser (구조화 텍스트 변환) |
| PDF | `.pdf` | PDFParser |
| Word | `.docx` | DocxParser |
| PowerPoint | `.pptx` | PptxParser |
| Excel | `.xlsx` | XlsxParser |
| 한글 | `.hwp` | HWPParser (한국 정부/관공서 문서) |

파서 선택: 확장자 기반 조회 우선, MIME 타입 감지(매직 바이트) 폴백.

### 청킹 전략

| 전략 | 사용 시점 | 설명 |
|------|----------|------|
| **auto** (기본) | 항상 | 문서 구조에 따라 자동 선택 |
| **structural** | 제목 감지 시 | Markdown/HTML 제목 기준 분할, 계층 구조 유지 |
| **semantic** | 단락 구조 | 의미 유사도 기반 그룹핑, 자연스러운 경계 존중 |
| **fixed** | 폴백 | 토큰 기반 분할, 문장 경계 존중 |

자동 선택 로직:
1. Markdown/HTML 제목 있음 → structural
2. 단락 구조 있음 (3개 이상 빈 줄) → semantic
3. 그 외 → fixed

### 4-Level Indexing

**Level 1: Multi-Resolution 요약**

더 거친 단위의 합성 청크 생성:
- **문서 수준**: 문서당 1개 — 전체 개요
- **섹션 수준**: H2/브레드크럼 그룹당 1개 — 섹션 요약
- **청크 수준**: 원본 청크 그대로

규칙 기반 요약 (LLM 비용 없음): 핵심 문장 추출, 엔티티/팩트 목록, 등급/상태 분포.

**Level 2: Structured Fact 추출**

규칙 기반 추출 (정규식, LLM 불필요) — 6개 도메인:

| 도메인 | 추출 팩트 | 예시 |
|--------|----------|------|
| 보안 | 이슈 ID, 등급, 상태 | `SEC-001`, `severity:Critical`, `status:완료` |
| 재무 | 지표, 자금 배분 | `매출 150억`, `R&D: 80억원` |
| 인사 | 팀 정보, 인원, 리더 | `AI팀 15명`, `팀장: 김철수` |
| 제품 | 버전, 릴리스, 변경사항 | `v2.4.0`, 릴리스 일자 |
| 특허 | 특허 ID, 발명자, 상태 | `PAT-003`, 발명자 목록 |
| 계약 | 고객사, 등급, 배포 방식 | `고객: A사`, `tier:Enterprise` |

팩트는 `chunk.metadata["facts"]`에 구조화된 딕셔너리로 저장.

**Level 3: Derived Index 보강**

인제스트 시점에 추가 BM25 검색 가능 용어 생성:

| 패턴 | 생성 용어 |
|------|----------|
| severity:Critical | "High 이상", "Medium 이상" (계층) |
| 일자 2024-07-20 | "하반기", "Q3", "2024년 Q3" (시간) |
| 금액 3.2억 | "약 3억", "억 단위" (정규화) |
| status:완료 | "조치 완료된", "해결됨" (동의어) |
| 버전 v2.4.0 | "v2.4", "2.4 버전" (정규화) |

**Level 4: Entity-Centric 역인덱스**

인메모리 역인덱스: `entity_key → set[chunk_id]`

인덱스 유형:
- 패턴 엔티티: `SEC-001`, `PAT-003`, `v2.5.0` (대문자 변환)
- 속성 엔티티: `severity:Critical`, `status:완료`, `tier:Enterprise`
- 등급 계층: Critical은 `severity_gte:High`, `severity_gte:Medium`에도 인덱싱
- 자금 배분: `type:fund_allocation`, `fund_item:{항목명}`
- 이름 엔티티: `person:{이름}`, `customer:{고객사명}`

### Triple Index

동일 청크에서 세 개의 병렬 검색 인덱스 구축:

| 인덱스 | 빌드 방식 | 포착 내용 |
|--------|----------|----------|
| **Original Embedding** | `chunk.content` 직접 임베딩 | 의미 유사도 |
| **HyPE Embedding** | 청크당 N개 가상 질문 생성 → 질문 임베딩 | 질문-질문 매칭 |
| **Contextual BM25** | 컨텍스트 접두사 추가 → Kiwi 토큰화 → 인덱싱 | 키워드 매칭 + 형태론 |

HyPE (Hypothetical Prompt Embedding)는 쿼리-문서 임베딩 간극을 해소합니다. 질문을 문서 패시지가 아닌 다른 질문과 비교하므로 유사한 임베딩 공간에서 매칭됩니다.

### Chunk Constellation Graph

사전 계산된 관계 네트워크:

| 엣지 유형 | 설명 | 가중치 |
|----------|------|--------|
| 형제 | 같은 문서의 인접 청크 | 1.0 |
| 부모 섹션 | 같은 제목/브레드크럼을 공유하는 청크 | 0.8 |
| 엔티티 교차 참조 | 같은 엔티티를 언급하는 청크 | 0.6 |

쿼리 시점에 BFS 탐색과 가중치 감쇠로 초기 결과를 관련 청크로 확장.

## 쿼리 파이프라인

쿼리 파이프라인은 사용자 질문을 출처가 있는 인용 답변으로 변환합니다.

### 단계별 흐름

```
사용자 쿼리
  │
  ▼
1. 대화 리라이트 ────── 히스토리 존재 시 대명사 해소
  │
  ▼
2. 쿼리 확장 ─────────── 구어체 → 정식 표현 리라이팅
  │
  ▼
3. 분류 ──────────────── SIMPLE / MEDIUM / COMPLEX 라우팅
  │
  ▼
4. 분해 ──────────────── 복합 질문을 서브 쿼리로 분리
  │
  ▼
5. 엔티티 감지 ──────── ID, 등급 필터, 상태 필터 파싱
  │
  ▼
6. Triple Index Fusion 검색
  │  ├─ Original Embedding 검색
  │  ├─ HyPE Embedding 검색
  │  └─ BM25 키워드 검색
  │  └─ Reciprocal Rank Fusion (RRF)
  │
  ▼
7. Entity Index 주입 ── 완전 재현율 보장 청크 병합
  │
  ▼
8. 형제 확장 ─────────── Constellation Graph 기반 관련 청크 추가
  │
  ▼
9. 리랭킹 ──────────── 크로스 인코더 정밀도 스코어링
  │
  ▼
10. MMR 다양성 ──────── 결과 내 중복 감소
  │
  ▼
11. 압축 ────────────── 추출형, 쿼리 인식 문장 선택
  │
  ▼
12. 생성 ────────────── 출처 기반 답변 생성 + [1][2] 인용
  │
  ▼
13. 자기 보정 ────────── 근거 부족 시 확장 검색으로 재시도
  │
  ▼
14. Map-Reduce ──────── 열거/집계 감지 시 병렬 집계
  │
  ▼
답변 + 출처 + 신뢰도
```

### 적응형 쿼리 라우팅

| 복잡도 | 트리거 | 모델 티어 | 파이프라인 |
|--------|--------|----------|-----------|
| **SIMPLE** | 짧은 사실 질문 | nano | Fusion → 생성 |
| **MEDIUM** | How/Why, 긴 쿼리, 시간 관련 | mini | Fusion → 리랭크 → 생성 |
| **COMPLEX** | 비교, 복합 질문, 조건부 | full | 분해 포함 전체 파이프라인 |

### Triple Index Fusion (RRF)

Reciprocal Rank Fusion으로 세 인덱스 결과 결합:

```
RRF_score(doc) = Σ (weight_i / (k + rank_i))
```

- `k = 60` (스무딩 상수)
- 가중치: Original 0.4, HyPE 0.35, BM25 0.25
- 후보: 인덱스당 `top_k × fusion_candidate_multiplier`개
- 최종 점수 [0, 1] 범위로 정규화

### 엔티티 주입

쿼리에서 엔티티 패턴 감지 시 (예: "SEC-001", "severity >= High"):

1. 엔티티 디텍터가 쿼리에서 제약 조건 파싱
2. 엔티티 역인덱스가 매칭 chunk ID 반환
3. **누락 청크**는 검색 결과에 주입
4. **압축된 청크**는 전체 콘텐츠로 교체 (추출 압축으로 인한 데이터 손실 방지)

### Map-Reduce RAG

집계 쿼리 감지 시 트리거 ("모두 나열", "전체 목록"):

1. 넓은 범위 검색 (20+ 청크)
2. **Map 단계**: 각 청크에서 관련 팩트를 독립적으로 추출
3. **Reduce 단계**: 추출된 팩트를 최종 답변으로 집계
4. `STRONGLY_SUPPORTED` 신뢰도 반환

### 자기 보정 RAG

초기 생성이 `INSUFFICIENT_EVIDENCE`를 반환하면:

1. 초기 답변에서 부족한 초점 추출
2. 초점 특화 쿼리 + 2배 `top_k`로 재시도
3. 원본 + 재시도 청크 병합
4. 확장된 컨텍스트로 재생성

## 스토리지 레이어

| 컴포넌트 | 백엔드 | 용도 |
|----------|--------|------|
| **Document Store** | SQLite | 문서, 청크, 메타데이터 |
| **Vector Store** | LanceDB | 임베딩 유사도 검색 |
| **BM25 Store** | Tantivy | 전문 키워드 검색 |

기본적으로 모두 로컬 (외부 서비스 불필요). 스토리지 백엔드는 팩토리 패턴을 통해 교체 가능.
