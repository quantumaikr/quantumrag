# QuantumRAG를 만든 이유

> 문서를 넣으세요. 질문하세요. 나머지는 알아서 합니다.

---

## 기존 RAG의 한계

RAG(Retrieval-Augmented Generation)는 LLM 기반 지식 시스템의 사실상 표준이 되었습니다. 문서를 벡터 DB에 저장하고, 질문을 임베딩으로 변환해 유사한 문단을 찾고, LLM에게 답변을 생성하게 하는 구조입니다. 직관적이고 구현도 간단합니다.

하지만 실제 업무 환경에 적용하면 이야기가 달라집니다.

### 임베딩 검색만으로는 정확한 답을 찾을 수 없다

"High 등급 이상 보안 이슈를 나열해주세요"라는 질문을 생각해보겠습니다. 이 질문에 대한 정답은 "SEC-003: 인증 모듈 SQL 인젝션, 등급 Critical"과 같은 개별 이슈 항목입니다. 그런데 임베딩 기반 유사도 검색에서 이 둘은 의미적으로 가까운 문장이 아닙니다. 임베딩은 "보안"이라는 주제는 잡아내지만, 등급 조건에 맞는 정확한 데이터를 골라내지는 못합니다.

숫자, 코드명, 상태값처럼 구조화된 조건이 포함된 질문일수록 이 문제는 심각해집니다.

### 한국어 환경이 고려되지 않았다

대부분의 RAG 프레임워크는 영어 중심으로 설계되어 있습니다. 한국어는 교착어라는 근본적인 차이가 있어서, 공백 기반 토큰화로는 BM25 키워드 검색이 제대로 동작하지 않습니다. "보안이슈의심각도가높습니다"라는 문장에서 "보안", "이슈", "심각도"를 분리하려면 형태소 분석이 필수입니다.

정부기관과 대부분의 기업이 사용하는 HWP 문서 형식은 지원되지 않고, EUC-KR 인코딩 파일은 깨져서 읽힙니다. 한국어 사용자에게 기존 RAG 프레임워크는 반쪽짜리 도구입니다.

### 단순한 파이프라인으로는 실전 질문에 답할 수 없다

실제 사용자의 질문은 간단하지 않습니다.

- 복합 질문: "투자금의 가장 큰 사용 용도는 무엇이고, 금액은 얼마인가요?"
- 조건부 검색: "조치가 완료된 High 등급 이상 보안 이슈의 ID를 모두 나열해주세요"
- 교차 검증: "이사회 보고서에 나온 고객 수가 영업 파이프라인 데이터와 일치하나요?"

임베딩 유사도로 상위 문단 몇 개를 가져와서 LLM에 던지는 방식으로는 이런 질문에 안정적으로 답할 수 없습니다. 여러 문서에 흩어진 정보를 모으고, 조건에 맞게 필터링하고, 누락 없이 집계하는 체계적인 접근이 필요합니다.

## 설계 철학

QuantumRAG의 핵심 약속은 단순합니다: **문서를 넣고, 질문하면, 출처와 함께 정확한 답이 나옵니다.**

이것이 가능한 이유는 Index-Heavy, Query-Light 아키텍처에 있습니다.

> **문서를 저장할 때 충분히 이해해두면, 질문에 답할 때 해야 할 일이 줄어든다.**

기존 RAG는 인덱싱 단계에서 임베딩 생성과 저장만 수행하고, 검색 품질을 높이기 위한 모든 노력을 쿼리 시점에 집중합니다. QuantumRAG는 이 관계를 뒤집습니다.

문서를 인제스트하는 시점에 다음을 미리 파악해둡니다:

- 문서에 포함된 엔티티(SEC-001, PAT-003)와 그 속성(등급, 상태, 금액)
- 청크 간의 관계(같은 문서의 인접 문단, 같은 엔티티를 언급하는 다른 문서)
- 이 청크로 답할 수 있는 질문의 형태
- 같은 의미를 다르게 표현한 동의어와 상위/하위 개념

이 사전 분석 덕분에 쿼리 시점에는 정확한 인덱스를 조회하는 것만으로 높은 재현율을 확보할 수 있습니다. "조치 완료된 High 이상 보안 이슈"라는 질문이 들어오면, 임베딩 유사도에 의존하지 않고 Entity Reverse Index에서 `severity_gte:High`와 `status:완료` 조건을 직접 조회합니다. 결과는 밀리초 단위로 반환되고, 누락은 없습니다.

## 핵심 기술

### 1. Triple Index Fusion — 세 가지 관점의 검색을 하나로

대부분의 RAG 시스템은 임베딩 유사도 하나에 검색 품질 전체를 의존합니다. QuantumRAG는 서로 다른 강점을 가진 세 개의 인덱스를 동시에 검색하고, 그 결과를 Reciprocal Rank Fusion(RRF)으로 통합합니다.

| 인덱스 | 역할 | 보완하는 약점 |
|--------|------|-------------|
| **Original Embedding** | 의미적으로 관련된 문단 검색 | — |
| **HyPE Embedding** | 질문 형태로 변환된 임베딩 간 비교 | 질문과 문서 사이의 임베딩 공간 불일치 |
| **Contextual BM25** | 정확한 키워드 및 형태소 매칭 | 임베딩이 놓치는 고유명사, 숫자, 코드 |

HyPE(Hypothetical Prompt Embedding)는 특히 주목할 만한 기법입니다. 인덱싱 시점에 각 청크에 대해 "이 내용으로 답할 수 있는 질문"을 LLM으로 미리 생성하고, 그 질문을 임베딩해둡니다. 사용자가 질문을 하면 원문이 아닌 사전 생성된 질문과 비교하게 되는데, 질문끼리는 문서 원문보다 훨씬 가까운 임베딩 공간에 위치하기 때문에 검색 정확도가 크게 향상됩니다.

세 인덱스의 결과는 가중치 0.4 / 0.35 / 0.25로 융합되며, 어느 하나의 인덱스만 사용할 때보다 일관되게 높은 성능을 보입니다.

### 2. 4-Level Indexing — 문서를 깊이 이해하는 네 겹의 인덱스

Triple Index가 "어떻게 검색할 것인가"의 문제를 해결한다면, 4-Level Indexing은 "무엇을 검색 가능하게 만들 것인가"의 문제를 해결합니다.

**Level 1: Multi-Resolution 요약**

문서 전체, 섹션, 개별 청크 세 가지 해상도로 요약을 생성합니다. "이 보고서의 핵심 내용이 뭔가요?"처럼 넓은 범위의 질문에는 문서 수준 요약이, 세부 사항을 묻는 질문에는 청크 수준 원본이 활용됩니다.

**Level 2: 구조화된 팩트 추출**

보안, 재무, 인사, 제품, 특허, 계약 6개 도메인에서 엔티티와 속성을 규칙 기반으로 추출합니다. LLM을 사용하지 않으므로 비용이 들지 않으면서도, "SEC-001의 등급은 Critical이고 상태는 완료"와 같은 구조화된 정보를 확보합니다.

**Level 3: 파생 검색어 생성**

같은 의미를 다양하게 표현하는 검색어를 자동 생성합니다.

- 등급 계층: `Critical` → "High 이상", "Medium 이상"으로도 검색 가능
- 시간 정규화: `2024-07-20` → "하반기", "Q3", "2024년 3분기"
- 금액 정규화: `3.2억` → "약 3억", "억 단위"
- 상태 동의어: `완료` → "조치 완료된", "해결됨"

이 용어들이 BM25 인덱스에 추가되므로, 사용자가 어떤 표현을 쓰더라도 관련 문서를 놓치지 않습니다.

**Level 4: Entity Reverse Index**

`SEC-001`, `severity:Critical`, `customer:A사` 같은 엔티티 키를 청크 ID에 직접 매핑하는 역인덱스입니다. 엔티티 기반 쿼리에 대해 100% 재현율을 보장합니다. 임베딩 유사도 점수가 낮아서 결과에서 빠지는 일이 구조적으로 발생할 수 없습니다.

### 3. 적응형 쿼리 라우팅 — 질문의 난이도에 맞는 최적 경로

모든 질문에 동일한 파이프라인을 적용하는 것은 비효율적입니다. 단순한 사실 확인에 최고 성능 모델과 전체 파이프라인을 동원할 이유가 없고, 복잡한 교차 분석에 경량 모델을 쓰면 품질이 떨어집니다.

QuantumRAG는 질문을 자동으로 분류하고, 복잡도에 맞는 처리 경로를 선택합니다.

| 복잡도 | 비율 | 처리 방식 |
|--------|:----:|----------|
| **단순** | ~70% | 경량 모델, 리랭킹 생략, 빠른 응답 |
| **중간** | ~20% | 리랭킹 적용, 확장 검색 |
| **복잡** | ~10% | 쿼리 분해, 엔티티 감지, Map-Reduce 집계, 고성능 모델 |

이 계층형 라우팅으로 전체 쿼리를 고성능 파이프라인에 통과시키는 것 대비 평균 비용을 60~70% 절감하면서도, 어려운 질문에서의 답변 품질은 유지합니다.

여기에 더해 상황별로 자동 활성화되는 고급 기능들이 있습니다:

- **엔티티 감지 & 역인덱스 조회**: 쿼리에 "SEC-001", "High 등급 이상" 같은 패턴이 포함되면 Entity Reverse Index를 통해 정확한 청크를 보장
- **쿼리 분해**: "A는 무엇이고 B는 얼마인가요?" 같은 복합 질문을 독립된 서브 쿼리로 분리하여 각각 검색
- **Map-Reduce**: "모든 항목을 나열해주세요" 같은 열거형 질문에 대해 다수의 청크에서 병렬로 정보를 추출한 뒤 하나로 합산
- **자기 보정**: 첫 번째 답변이 근거 부족이면 검색 범위를 자동으로 넓혀 재시도
- **확장 검색 트리거**: 비교, 최상급, 다중 조건 질문은 자동으로 검색 범위를 확대

### 4. 한국어 네이티브 지원

QuantumRAG는 한국어를 번역이 아닌 모국어로 대합니다. 설계 단계부터 한국어를 핵심 언어로 고려했습니다.

**형태소 분석 기반 검색.** Kiwi 형태소 분석기가 BM25 인덱싱과 검색 양쪽에 적용됩니다. "보안이슈의심각도가높습니다"처럼 조사와 어미가 붙어 하나의 덩어리가 되는 한국어 텍스트에서 "보안", "이슈", "심각도"를 정확히 분리해야 키워드 검색이 작동합니다.

**HWP/HWPX 네이티브 파싱.** 공공기관 문서, 사내 보고서 등 한국 업무 환경에서 가장 많이 사용되는 한글 문서 형식을 별도 변환 없이 직접 파싱합니다.

**한국어 쿼리 패턴 인식.** "High 등급 이상", "조치 완료된", "모두 나열해주세요" 같은 한국어 고유의 질문 패턴을 형태소 수준에서 인식하고, 적절한 검색 전략으로 연결합니다.

**표현 다양성 대응.** "Critical"을 "심각", "높음 등급", "High 이상"으로도 검색할 수 있도록, 인덱싱 시점에 한국어 동의어와 계층 용어를 자동으로 생성합니다.

**한국어 특화 생성 규칙.** 답변 생성에 11가지 한국어 전용 규칙을 적용합니다. 경쟁사와 자사 정보의 혼동 방지, 복수 출처의 수치 불일치 시 각각의 수치와 출처 병기 등, 한국어 기업 문서에서 자주 발생하는 문제를 체계적으로 처리합니다.

### 5. 프로덕션 환경을 위한 인프라

QuantumRAG는 프로토타입이 아니라 실서비스를 고려한 엔진입니다.

- **FastAPI 기반 HTTP API**: SSE 스트리밍, API 키 인증, 요청 제한, 경로 접근 보호
- **플러그인 시스템**: 파서, 청커, 검색기, 생성기를 코어 수정 없이 확장
- **멀티 테넌시**: 테넌트별 격리된 저장소, 개별 모델 설정, 사용량 쿼터
- **시맨틱 캐시**: 동일하거나 유사한 질문에 대해 LLM 재호출 없이 캐시된 결과 반환
- **구조화된 로깅과 트레이싱**: 쿼리별 파이프라인 실행 추적, SQLite 영구 저장
- **5종 데이터 커넥터**: 로컬 파일, Google Drive, Notion, AWS S3, 웹 URL

### 6. 외부 인프라 없이 즉시 실행

QuantumRAG는 외부 서비스 의존 없이 로컬 환경에서 완전하게 동작합니다.

| 컴포넌트 | 역할 |
|---------|------|
| SQLite | 문서 및 메타데이터 저장 |
| LanceDB | 벡터 유사도 검색 |
| Tantivy | BM25 전문 검색 |
| FlashRank | CPU 기반 리랭킹 (무료) |
| Ollama | 로컬 LLM 추론 (선택) |

Docker, Kubernetes, 관리형 벡터 DB, GPU — 모두 필요 없습니다. `pip install quantumrag[all]` 한 줄이면 바로 시작할 수 있습니다.

## 검증

QuantumRAG는 16개 카테고리, 4단계 난이도(Easy~Extreme)로 구성된 **87개 실전 시나리오 테스트**에서 **86/87 통과 (98.9%)** 를 기록하고 있습니다.

테스트에는 일반적인 RAG 시스템이 실패하기 쉬운 항목들이 포함되어 있습니다:

- 3개 이상의 문서에 걸친 멀티홉 추론
- 여러 위치에 분산된 수치를 조합하는 계산
- 등급, 상태 조건을 결합한 엔티티 필터링
- 명시되지 않은 정보에 대한 추론
- 문서 간 수치 일관성 교차 검증
- 적대적 입력 및 경계 조건 처리

모든 테스트는 동일한 엔진 설정으로 실행되며, 키워드 매칭 기반의 자동화된 검증을 사용합니다.

## 세 가지 사용 방식

QuantumRAG는 설정 없이 시작할 수 있고, 필요할 때 모든 것을 제어할 수 있습니다.

| | 하는 일 | 일어나는 일 |
|---|---------|-----------|
| **그냥 쓰면** | `engine.ingest("./docs")` → `engine.query("...")` | 파서, 청커, 인덱스, 라우팅 — 모두 자동 |
| **좀 더 쓰면** | 퓨전 가중치 조정, 모델 선택, 도메인 설정 | 특정 유스케이스에 더 좋은 결과 |
| **깊이 쓰면** | 커스텀 파서, 청커, 리트리버, 제너레이터 | 모든 레이어를 플러그인으로 교체 가능 |

## 이런 분들을 위해 만들었습니다

**기업 지식 관리 담당자** — 이사회 의사록, 감사 보고서, 계약서, 특허 문서, 제품 변경 이력 등 한국어 기업 문서 위에 사내 지식 검색 시스템을 구축해야 하는 분.

**한국어 RAG를 구축하는 개발자** — 형태소 분석기 연동, HWP 파서 통합, 한국어 프롬프트 최적화에 시간을 쏟지 않고 빠르게 한국어 RAG를 구축하고 싶은 분.

**정밀한 검색이 필요한 팀** — "특정 엔티티의 특정 속성" 수준의 정확한 검색과 자유로운 자연어 검색을 하나의 시스템에서 모두 지원해야 하는 분.

**RAG 전문가가 아닌 개발자** — LangChain 체인을 조립하거나 LlamaIndex 파이프라인을 설계할 시간 없이, 문서를 넣고 질문하면 바로 작동하는 엔진이 필요한 분.

## 시작하기

```bash
pip install quantumrag[all]
```

```python
from quantumrag import Engine

engine = Engine()
engine.ingest("./docs")
result = engine.query("Triple Index Fusion은 어떻게 동작하나요?")
print(result.answer)
```

세 줄이면 됩니다. 나머지는 알아서 합니다.
