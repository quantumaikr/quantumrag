# QuantumRAG 소개

> 왜 새로운 RAG 엔진을 만들었는가, 그리고 무엇이 다른가.

---

## 문제 인식

검색 증강 생성(RAG)은 LLM 기반 지식 시스템을 구축하는 표준 접근법이 되었습니다. 문서를 벡터 데이터베이스에 넣고, 사용자의 질문을 임베딩하고, 유사한 청크를 검색하고, LLM에 전달하여 답변을 생성합니다. 간단하고, 우아하고 — 그리고 실제 프로덕션 환경에서는 심각하게 부족합니다.

프로덕션에서 무엇이 깨지는가:

**임베딩만으로는 핵심 정보를 놓칩니다.** "High 등급 이상 보안 이슈"라는 질문은 "SEC-003: 인증 모듈에서 SQL 인젝션 취약점 발견, 등급 Critical"이라는 청크와 의미적 유사도가 거의 없습니다. 임베딩 검색은 주제적으로 관련된 문서를 찾지만, 질문에 답하기 위해 필요한 정확한 데이터를 찾지는 못합니다.

**한국어는 뒷전입니다.** 대부분의 RAG 프레임워크는 한국어를 "그저 또 하나의 언어"로 취급합니다. 그러나 한국어의 교착어 형태론은 적절한 토큰화 없이는 BM25 키워드 검색이 무용지물이 됩니다. 한글 문서 형식 HWP — 사실상 모든 한국 정부기관과 기업이 사용하는 — 는 지원되지 않습니다. EUC-KR 인코딩된 레거시 파일은 조용히 깨집니다.

**쿼리 시점 연산이 병목입니다.** 표준 RAG 패턴은 인덱싱 시점에 최소한의 작업(임베딩과 저장)만 수행하고, 모든 지능을 쿼리 시점(검색, 리랭킹, 생성)에 밀어넣습니다. 이는 매 쿼리마다 문서 컬렉션을 처음부터 이해하는 전체 비용을 지불한다는 의미입니다.

**단순 파이프라인은 실제 질문에 실패합니다.** 실제 사용자는 복합 질문("투자금의 가장 큰 사용 용도는 무엇이고 얼마인가요?"), 엔티티 특정 쿼리("조치 완료된 High 등급 이상 보안 이슈를 모두 나열"), 교차 문서 질문("이사회 보고서의 고객 수가 영업 파이프라인 데이터와 일치하는가?")을 합니다. 단일 임베딩 검색 후 LLM 생성으로는 이 중 어느 것도 안정적으로 답할 수 없습니다.

---

## 설계 철학

QuantumRAG는 하나의 통찰에서 출발했습니다: **인덱싱 시점에 문서를 깊이 이해할수록, 쿼리 시점에 해야 할 일이 줄어든다.** 이것을 **Index-Heavy, Query-Light** 라 부릅니다.

문서를 임베딩할 불투명한 텍스트로 취급하는 대신, QuantumRAG는 인제스트 시점에 모든 문서에 대한 풍부하고 다층적인 이해를 구축합니다:

- 이 문서에는 어떤 엔티티와 팩트가 포함되어 있는가?
- 핵심 속성은 무엇인가 — 등급, 상태값, 재무 수치, 버전 번호?
- 청크들은 서로 어떤 관계인가 — 형제, 부모 섹션, 공유 엔티티?
- 이 청크가 답할 수 있는 질문은 무엇인가?
- 이 콘텐츠를 찾는 데 도움이 될 동의어와 계층 용어는 무엇인가?

이 선행 투자는 즉시 보상을 돌려줍니다. 사용자가 "조치 완료된 Critical, High 등급 보안 이슈를 모두 나열해주세요"라고 물으면, 엔진은 임베딩 유사도가 적절한 청크를 찾아주길 바랄 필요가 없습니다. Entity Reverse Index에서 `severity_gte:High`와 `status:완료`를 조회하여 정확한 청크를 반환합니다 — 100% 재현율, 밀리초 단위.

## QuantumRAG의 차별점

### 1. Triple Index Fusion

대부분의 RAG 시스템은 단일 검색 방법 — 보통 임베딩 유사도 — 을 사용합니다. QuantumRAG는 세 개의 인덱스를 동시에 검색하고 결과를 융합합니다:

| 인덱스 | 포착 내용 | 중요한 이유 |
|--------|----------|------------|
| **Original Embedding** | 의미적 유사성 | 주제적으로 관련된 문서 발견 |
| **HyPE Embedding** | 질문 수준 의미론 | 질문-질문 비교로 쿼리-문서 간극 해소 |
| **Contextual BM25** | 정확한 키워드 + 형태론 | 임베딩이 놓치는 엔티티, 숫자, 기술 용어 포착 |

세 결과 집합은 튜닝된 가중치(0.4 / 0.35 / 0.25)의 Reciprocal Rank Fusion (RRF)으로 결합됩니다. 이 하이브리드 접근법은 어떤 단일 인덱스보다도 일관되게 우수한 성능을 보입니다.

**HyPE (Hypothetical Prompt Embedding)**는 특별히 주목할 가치가 있습니다. 인덱싱 시점에 엔진은 각 청크에 대해 가상 질문을 생성합니다 — "SEC-003에서 발견된 보안 취약점은?", "인증 모듈 이슈의 등급은?" — 그리고 이 질문들을 임베딩합니다. 쿼리 시점에 사용자의 질문은 원시 문서 텍스트가 아닌 사전 생성된 질문과 비교됩니다. 질문과 질문은 질문과 문서보다 훨씬 유사한 임베딩 공간에 존재하므로, 질의응답 워크로드의 검색 성능이 극적으로 향상됩니다.

### 2. 4-Level Indexing

Triple Index 너머, QuantumRAG는 인제스트 시점에 네 가지 추가적인 이해 레이어를 구축합니다:

**Level 1: Multi-Resolution 요약.** 각 문서는 세 가지 단위 — 문서 수준, 섹션 수준, 청크 수준 — 로 요약됩니다. 사용자가 넓은 질문("이 보고서가 무엇에 관한 건가요?")을 하면, 문서 수준 요약이 개별 청크를 이어 붙일 필요 없이 직접적인 답변을 제공합니다.

**Level 2: Structured Fact 추출.** 규칙 기반 추출 엔진이 6개 도메인 — 보안, 재무, 인사, 제품, 특허, 계약 — 에 걸쳐 엔티티, 속성, 관계를 식별합니다. 이 구조화된 팩트는 임베딩 검색만으로는 불가능한 정밀한 필터링을 가능하게 합니다.

**Level 3: Derived Index 보강.** 인덱싱 시점에 추가 검색 가능 용어를 생성합니다: 등급 계층("Critical" → "High 이상"으로도 검색 가능), 시간 정규화(날짜 → "Q3", "하반기"), 금액 정규화, 상태 동의어. 이 용어들은 BM25 인덱스에 주입되어, 다른 표현의 한국어 쿼리도 올바른 콘텐츠를 찾을 수 있게 합니다.

**Level 4: Entity-Centric 역인덱스.** 인메모리 역인덱스가 엔티티 키(`SEC-001`, `severity:Critical`, `customer:Acme`)를 청크 ID에 직접 매핑합니다. 엔티티 기반 쿼리에 대해 **100% 재현율을 보장**합니다 — 임베딩 유사도 임계값으로 인한 누락이 발생할 수 없습니다.

### 3. 적응형 쿼리 인텔리전스

모든 질문에 같은 파이프라인이 필요하지 않습니다. QuantumRAG는 쿼리를 복잡도 등급으로 분류하고 동작을 적응시킵니다:

- **단순 질문** (~70%)은 경량 모델을 사용하고, 리랭킹을 건너뛰고, 빠르게 반환
- **중간 질문** (~20%)은 리랭킹과 넓은 범위 검색을 추가
- **복잡한 질문** (~10%)은 완전한 분해, 엔티티 감지, Map-Reduce 집계, 최고 성능 모델을 사용

이 계층형 접근법은 모든 것을 최대 성능 파이프라인으로 라우팅하는 것 대비 평균 비용을 60-70% 절감하면서, 필요한 곳에서 품질을 유지합니다.

추가 적응형 동작:

- **엔티티 감지**: 쿼리에 엔티티 패턴("SEC-001", "severity >= High", "status:완료")이 포함되면, 보장된 재현율을 위해 Entity Reverse Index를 활성화
- **쿼리 분해**: 복합 질문을 독립 서브 쿼리로 분리, 각각 자체 컨텍스트를 검색
- **Map-Reduce RAG**: 열거 쿼리("모두 나열", "전체 목록")는 다수의 청크에 대한 병렬 추출 후 집계를 트리거
- **자기 보정 RAG**: 초기 생성이 `INSUFFICIENT_EVIDENCE`를 반환하면, 엔진이 자동으로 더 넓은 검색으로 재시도
- **넓은 범위 검색 패턴**: 최상급, 비교, 다중 조건 쿼리는 자동으로 검색 범위를 확장

### 4. 한국어를 1등 시민으로

QuantumRAG는 한국어를 플러그인으로 덧붙인 것이 아니라 처음부터 한국어를 기반으로 설계되었습니다:

- **Kiwi 형태소 분석**이 BM25 인덱싱을 구동합니다. 한국어는 교착어입니다 — "보안이슈의심각도가높습니다"는 공백 기준 하나의 "단어"이지만 네 개의 의미 있는 형태소를 포함합니다. 적절한 토큰화 없이는 한국어 BM25 검색이 무용지물입니다.
- **HWP/HWPX 파싱**이 한국 정부기관과 기업의 표준 문서 형식을 네이티브하게 처리합니다.
- **한국어 쿼리 패턴**이 등급 범위("High 등급 이상"), 상태 필터("조치 완료된"), 열거 요청("모두 나열해주세요")을 형태론 인식 정규식으로 인식합니다.
- **Derived 한국어 용어**가 "Critical"을 "심각", "High 이상", "높음 등급"으로 인제스트 시점에 동의어를 생성하여 검색 가능하게 합니다.
- **이중 언어 시스템 프롬프트**의 11가지 전문 한국어 생성 규칙이 경쟁사 엔티티 귀속, 다중 출처 수치 불일치 등 엣지 케이스를 처리합니다.

### 5. 프로덕션 레디 인프라

QuantumRAG는 노트북 데모가 아닙니다. 실제 배포에 필요한 인프라를 포함합니다:

- **FastAPI 서버** — SSE 스트리밍, API 키 인증, 레이트 리미팅, 경로 탐색 방어
- **플러그인 시스템** — 코어를 포크하지 않고 모든 파이프라인 단계 확장
- **멀티 테넌시** — 격리된 스토리지, 테넌트별 모델, 쿼터 적용
- **시맨틱 캐시** — 정확/코사인 유사도 매칭으로 중복 LLM 호출 방지
- **구조화된 관측성** — structlog 기반 쿼리별 트레이싱, SQLite 저장
- **5개 데이터 커넥터** — 로컬 파일, Google Drive, Notion, AWS S3, 웹 URL
- **내장 평가** — 6개 메트릭, 합성 QA 생성, A/B 설정 비교

### 6. 외부 인프라 불필요

QuantumRAG는 외부 서비스 없이 완전히 로컬에서 실행됩니다:

- **SQLite** — 문서 저장
- **LanceDB** — 벡터 검색
- **Tantivy** — BM25 전문 검색
- **FlashRank** — CPU 기반 리랭킹 (무료, GPU 불필요)
- **Ollama** — 로컬 LLM 추론 (선택)

Docker 없음, Kubernetes 없음, 관리형 벡터 데이터베이스 없음, GPU 없음. `pip install quantumrag[all]`이면 바로 실행됩니다.

## 검증된 결과

QuantumRAG는 4단계 난이도(Easy~Extreme), 16개 카테고리에 걸친 **87개 실전 시나리오 테스트**로 검증되었습니다. 현재 통과율: **86/87 (98.9%)**.

테스트 스위트에는 대부분의 RAG 시스템이 실패하는 도전적인 항목이 포함됩니다:

- 3개 이상 문서에 걸친 멀티홉 추론
- 산재된 데이터 포인트에서의 수치 계산
- 등급 및 상태 제약 조건이 있는 엔티티 특정 쿼리
- 직접 명시되지 않은 정보로부터의 암묵적 추론
- 교차 문서 일관성 검증
- 적대적 및 엣지 케이스 쿼리

이것들은 선별된 예시가 아닙니다. 전체 테스트 스위트는 매번 동일한 엔진 설정으로 실행되며, 키워드 매칭 검증을 사용합니다 — 사람의 채점이 아닙니다.

## 누구를 위한 것인가

**기업 팀** — 한국어와 영어 기업 문서(이사회 의사록, 감사 보고서, 계약서, 특허 출원, 제품 변경 이력) 위에 내부 지식 시스템을 구축하는 팀.

**개발자** — 형태소 분석 통합과 HWP 파싱에 몇 주를 쓰지 않고, 한국어에 즉시 작동하는 RAG 엔진이 필요한 개발자.

**팀** — 두 개의 별도 시스템을 구축하지 않고, 구조화 데이터 정밀도(엔티티 조회, 속성 필터링)와 시맨틱 검색의 유연성을 결합하고 싶은 팀.

**누구나** — RAG 데모 품질과 프로덕션 RAG 품질 사이의 간극에 좌절한 모든 사람.

## 시작하기

```bash
pip install quantumrag[all]
```

```python
from quantumrag import Engine

engine = Engine()
engine.ingest("./docs")
result = engine.query("매출이 얼마인가요?")
print(result.answer)
```

세 줄. 설정 불필요. 그냥 됩니다.
