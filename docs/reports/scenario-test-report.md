## 4. 핵심 기능 영향 분석

### 4.1 Triple Index Fusion

검색 정확도의 근간이 되는 세 가지 인덱스 융합 전략:

| 인덱스 | 가중치 | 효과 확인 시나리오 |
|--------|--------|-------------------|
| Original Embedding (벡터) | 0.40 | S1(사실), S7(패러프레이즈) — 의미 유사도 기반 검색 |
| HyPE Embedding (가상 질문) | 0.35 | S4(시간), S10(정밀) — 질문-질문 매칭으로 정밀도 향상 |
| Contextual BM25 (키워드) | 0.25 | S3(수치), S14(필터링) — 정확한 수치/이름 매칭 |

### 4.2 Contextual Chunk Enrichment

- 각 청크에 문서 계층(breadcrumb) 자동 주입
- **효과**: S4(시간/버전 추론)에서 버전 번호-기능 매핑 정확도 향상
- 예: "v2.5.0 > 신기능 > HyPE" breadcrumb → 버전별 기능 질문에 정확 응답

### 4.3 Entity Memory Tracker

- 대화 중 엔티티를 유형별(회사/제품/인물)로 추적
- **효과**: S8(멀티턴)에서 6/6 통과
  - Turn 2: "그 제품" → QuantumGuard (제품 타입)
  - Turn 5: "그 회사" → 퀀텀아이 (회사 타입, 가장 최근 언급)

### 4.4 Self-Corrective RAG

- 첫 응답이 `insufficient_evidence`일 때 더 넓은 검색으로 자동 재시도
- **발동 확인**: S4.2(HyPE 도입 전 검색 방식), S9.2/9.6/9.7(문서에 없는 내용)
- 로그에서 `self_correct_triggered` 이벤트 5회 확인

### 4.5 Map-Reduce RAG

- 열거/집계 질문에서 다수 청크를 병렬 처리
- **발동 확인**: S2.5(일본 PoC 3건 합산), S3.3(개발 부서 예산 합계), S5.2(멀티모달 미지원 전체), S7.4(일본 최대 계약)
- 로그에서 `map_reduce_reducing` 이벤트 확인 (최대 45개 청크 병렬 처리)

### 4.6 Query Expander

- 구어체/비격식 질문을 정규화된 검색 키워드로 확장
- **발동 확인**:
  - "해킹 당하면 제일 위험한 게 뭐에요?" → "사이버 공격 시 가장 치명적 위험 데이터 유출"
  - "우리 기술 뺏기지 않으려고 뭐 해놨어?" → "기술 보호 지식재산권 영업비밀 특허 전략"
  - "일본에서 제일 큰 계약이 될 곳은 어디야?" → "일본 최대 규모 계약 대형 프로젝트 발주처"

## 6. 실패 분석 및 개선 제안

### 6.1 유일한 실패 케이스 [12.3]

**질문**: "42Maru와 NTT 데이터 PoC에서 경쟁하고 있는데, 42Maru 대비 퀀텀소프트의 기술적 장점은?"

**기대**: "Triple Index", "한국어", "검색 정확도" 중 하나 이상 포함

**실제 응답**: "직접 비교해 단정하기에 정보가 부족합니다" → 간접적으로 기술 차이를 설명했으나 기대 키워드를 명시적으로 포함하지 않음

**원인 분석**:
1. **Retrieval 성공**: Comparative Decomposition이 발동되어 42Maru와 퀀텀소프트를 각각 검색
2. **Generation 보수적 판단**: LLM이 "NTT PoC 맥락에서의 직접 비교"는 문서에 없다고 판단
3. **키워드 매칭 한계**: 답변에 "기술 특징"으로 간접 언급했으나, 정확히 "Triple Index"라는 키워드가 포함되지 않음

**개선 방안**:
- Generation 프롬프트에서 비교 질문 시 문서 내 기술 비교표를 적극 인용하도록 개선
- 또는 테스트 기대 키워드를 "MRC", "LLM 전환" 등 42Maru 약점 키워드로 확대

### 6.2 향후 개선 방향

| 영역 | 현재 상태 | 개선 방향 |
|------|----------|----------|
| FlashRank Reranker | 미설치 (6회 경고) | `pip install flashrank`로 설치 시 정확도 추가 향상 기대 |
| Comparative Query | 규칙 기반 분해 | LLM 기반 비교 관점 자동 생성 |
| 인제스트 속도 | 390초 (15문서) | HyPE 배치 처리 최적화로 50% 단축 목표 |
| GraphRAG | Chunk Graph 기본 | 엔티티 관계 그래프로 멀티홉 근본 개선 |
