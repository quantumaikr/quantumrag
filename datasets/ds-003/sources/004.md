# Gang of Four Design Patterns

## Overview

"Design Patterns: Elements of Reusable Object-Oriented Software" was published by four authors: Erich Gamma, Richard Helm, Ralph Johnson, and John Vlissides — known as "The Gang of Four" (GoF). These patterns provide a standardized approach to solving recurring design problems in object-oriented software.

## Creational Design Patterns

Creational patterns focus on object creation mechanisms, providing flexibility in what gets created, who creates it, how it gets created, and when.

- **Abstract Factory**: Creates object families without specifying concrete types. Provides an interface for creating families of related or dependent objects without specifying their concrete classes.

- **Builder**: Constructs complex objects step-by-step. Separates construction from representation, allowing the same construction process to create various representations.

- **Factory Method**: Creates objects without specifying exact classes. Defines an interface for creating an object but lets subclasses alter the type of objects that will be created.

- **Prototype**: Creates new objects by copying existing ones. Used to create a new object by copying an existing one, which can then be modified as needed.

- **Singleton**: Ensures only one object instance exists. Restricts the instantiation of a class to a single object with global access.

## Structural Design Patterns

Structural patterns define relationships between objects, composing them into larger structures while maintaining flexibility.

- **Adapter**: Enables incompatible classes to work together. Wraps an interface around one of the existing classes, useful for integrating legacy systems.

- **Bridge**: Decouples abstraction from implementation. Separates the interface from the implementation, ensuring that modifications to either aspect do not affect the other.

- **Composite**: Combines objects into tree structures to represent part-whole hierarchies. Clients treat individual objects and compositions uniformly.

- **Decorator**: Extends object behavior dynamically at runtime. Adds new responsibilities to objects without modifying their structure, offering flexible alternatives to subclassing.

- **Facade**: Simplifies complex subsystem interactions. Provides a simple interface to a more complex underlying object or set of objects.

- **Flyweight**: Reduces costs of complex object models. Manages large numbers of similar objects efficiently by sharing common data (intrinsic state) and keeping variable data (extrinsic state) external.

- **Proxy**: Controls access to underlying objects. Provides a placeholder interface to an underlying object to control access, reduce cost, or reduce complexity. Common uses include lazy initialization, logging, and access control.

## Behavioral Design Patterns

Behavioral patterns address algorithms and object responsibility assignment, managing complex control flows.

- **Chain of Responsibility**: Delegates commands to a chain of processing objects. Each object in the chain either handles the request or passes it to the next handler. Avoids coupling the sender to its receiver.

- **Command**: Encapsulates actions and parameters as objects. Promotes loose coupling in request-response models. Supports undoable operations and macro recording.

- **Interpreter**: Implements specialized languages by defining grammatical representations and an interpreter to process them.

- **Iterator**: Accesses collection elements sequentially without exposing underlying representation. Offers a standard way to traverse through a group of objects.

- **Mediator**: Provides centralized communication between objects. Allows loose coupling between classes by being the only class that has detailed knowledge of their methods.

- **Memento**: Captures and restores object state. Provides the ability to restore an object to its previous state without violating encapsulation. Useful for implementing undo functionality.

- **Observer**: Implements publish/subscribe mechanisms. Allows a number of observer objects to see an event, enabling one-to-many dependency relationships.

- **State**: Alters object behavior when internal state changes. The object appears to change its class.

- **Strategy**: Selects algorithms on-the-fly at runtime. Allows one of a family of algorithms to be selected on-the-fly at runtime. Defines a family of algorithms, encapsulates each one, and makes them interchangeable.

- **Template Method**: Defines algorithm skeletons in abstract classes. Creates a template method stub and defers some steps to subclasses, allowing subclasses to redefine certain steps without changing the algorithm's structure.

- **Visitor**: Separates algorithms from object structures. Allows adding new operations to existing object structures without modifying them.
