# Retrieval Augmented Generation (RAG) for LLMs

## Introduction to RAG

RAG is a system that takes input and retrieves a set of relevant/supporting documents given a source (e.g., Wikipedia). The documents are concatenated as context with the original input prompt and fed to a text generator which produces the final output. This approach makes RAG adaptive for situations where facts evolve over time, since LLMs' parametric knowledge is static.

The retrieved evidence enhances the accuracy, controllability, and relevancy of the LLM's response, helping reduce hallucination issues and improving performance in rapidly evolving environments.

### RAG Application Workflow

1. **Input:** The question the LLM system responds to
2. **Indexing:** Related documents are chunked, embeddings are generated, and indexed into a vector store
3. **Retrieval:** Relevant documents are obtained by comparing the query against indexed vectors
4. **Generation:** Retrieved documents combine with the original prompt as additional context

## RAG Paradigms

### Naive RAG

The traditional approach following indexing, retrieval, and generation. A user input queries relevant documents which are combined with a prompt and passed to the model.

Limitations:
- Low precision (misaligned retrieved chunks)
- Low recall (failure to retrieve all relevant chunks)
- Potential outdated information
- Hallucination issues
- Redundancy when using multiple passages
- Generation overly dependent on augmented information

### Advanced RAG

Addresses Naive RAG limitations through:

**Pre-retrieval:** Enhancing data granularity, optimizing index structures, adding metadata, alignment optimization, mixed retrieval.

**Retrieval stage:** Fine-tuning embedding models, employing dynamic embeddings for better contextual understanding.

**Post-retrieval:** Re-ranking to avoid context window limits, removing noisy information, prompt compression.

### Modular RAG

Enhances functional modules with greater diversity and flexibility. You can add or replace modules or adjust the flow between modules based on task requirements.

Extended modules:
- Search (similarity retrieval)
- Memory
- Fusion
- Routing
- Predict
- Task adapter

Optimization techniques:
- **Hybrid Search**: Combines keyword-based and semantic search
- **Recursive Retrieval**: Starts with small semantic chunks, then retrieves larger context-rich chunks
- **StepBack-prompt**: Abstraction that produces concepts and principles guiding reasoning
- **Sub-Queries**: Breaks queries into multiple questions for different data sources
- **HyDE (Hypothetical Document Embeddings)**: Generates hypothetical answers, embedding them for document retrieval

## RAG Framework Components

### Retrieval

#### Enhancing Semantic Representations

**Chunking:** Different models perform better with varying block sizes. Sentence transformers perform better on single sentences, but text-embedding-ada-002 performs better with blocks containing 256 or 512 tokens.

**Fine-tuned Embedding Models:** For specialized domains, fine-tuning embedding models may be required. BGE-large-EN by BAAI is a significant embedding model that can be fine-tuned to optimize retrieval relevance.

#### Aligning Queries and Documents

- **Query Rewriting:** Query2Doc, ITER-RETGEN, HyDE
- **Embedding Transformation:** Optimizes query embedding representations

#### Aligning Retriever and LLM

- **Fine-tuning Retrievers:** AAR, REPLUG, UPRISE
- **Adapters:** PRCA, RECOMP, PKG

### Generation

**Post-retrieval with Frozen LLM:** Information compression and result reranking without modifying the LLM.

**Fine-tuning LLM for RAG:** Optimize the generator for natural text and effective use of retrieved documents.

### Augmentation

#### Augmentation Stages
- **Pre-training:** RETRO leverages retrieval for large-scale pre-training
- **Fine-tuning:** Combines RAG with fine-tuning
- **Inference:** Various techniques for incorporating retrieved content

#### Augmentation Process

**Iterative Retrieval:** Multiple retrieval cycles to enhance depth and relevance. Approaches: RETRO, GAR-meets-RAG.

**Recursive Retrieval:** Uses output of one retrieval step as input to another. Approaches: IRCoT, Tree of Clarifications.

**Adaptive Retrieval:** Tailors retrieval to specific demands, determining optimal moments and content. Approaches: FLARE, Self-RAG.

## RAG vs. Fine-tuning

RAG is useful for integrating new knowledge while fine-tuning improves model performance through internal knowledge enhancement. These approaches are not mutually exclusive and can complement each other.

- RAG excels at accessing quickly-evolving knowledge
- Fine-tuning improves internal knowledge and response formatting
- Both can work together for complex knowledge-intensive applications

## RAG Evaluation

Three primary quality scores:
1. **Context Relevance:** Precision and specificity of retrieved context
2. **Answer Faithfulness:** Faithfulness of answers to retrieved context
3. **Answer Relevance:** Relevance of answers to posed questions

Four measurable abilities:
1. Noise robustness
2. Negative rejection
3. Information integration
4. Counterfactual robustness

Evaluation frameworks:
- RaLLe, RGB, RECALL, RAGAS, ARES, TruLens

## Challenges & Future

- Context length adaptation as LLMs extend windows
- Robustness to counterfactual and adversarial information
- Hybrid approaches combining RAG and fine-tuning
- Multimodal RAG beyond text (image, audio, video, code)
- Production-ready RAG for data security and privacy
- Developing better evaluation metrics
